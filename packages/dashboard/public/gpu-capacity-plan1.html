EXECUTIVE SUMMARY

GPU 용량 측정 방안

1 / 4

GPU 내부 메트릭(nvidia-smi, vLLM) 없이 기존 요청 로그만으로 GPU 사용률 / 부족분 / 미래 수요를 산출합니다.
즉시 적용 가능한 안 1을 먼저 도입하고, vLLM /metrics 개방 시점에 안 2로 고도화합니다.

배경

🎯 문제

현재 GPU 인프라의 실제 사용률을 정량적으로 파악할 수 없어,
"GPU가 부족한가? 얼마나 부족한가? 언제 증설해야 하는가?"에
데이터 기반 답변이 불가능한 상황입니다.

💡 제안

이미 수집 중인 요청 로그(request_logs)의 응답 시간 변화를 분석하여
GPU 사용률을 0~100%로 실시간 측정하고,
포화 전 조기 경보 + 미래 예측까지 제공합니다.

2단계 로드맵

Phase 1 — 즉시 적용
안 1. 응답 속도 변화 감지
~2일
P95 레이턴시의 baseline 대비 변화율로 사용률 0~100% 산출.
추가 인프라 없이 기존 로그만으로 포화 전 조기 경보 가능.
Phase 2 — vLLM /metrics 개방 후
안 2. 대기행렬 실측 기반
향후
현재 산술 추정만 가능하나, vLLM /metrics에서 실제 대기 인원 수를 읽으면
"GPU 정확히 N대 부족"을 수학적으로 정확하게 산출 가능.
📌
왜 안 1 먼저? 안 2의 대기행렬 이론은 정확하지만, 현재는 동시 요청 수·대기 시간을 로그에서 추정할 수밖에 없어 결국 추정치의 추정치가 됩니다. 안 1은 같은 데이터로 더 직관적인 결과를 내며, vLLM 실측이 가능해지면 안 2가 압도적으로 정확해집니다.
🚗
핵심 아이디어: 새벽에 20분 걸리던 출근길이 40분 걸리면 도로 50% 혼잡.
GPU도 "한산할 때 대비 얼마나 느려졌는가"로 포화도를 측정합니다.

구현 단계

1

선행 수정: 스트리밍 요청 분리 + Baseline 방식 변경

stream=true 제외(latency 부풀림), baseline = 최근 30일 RPS 하위 5% 시간대 P95

2

사용률 산출 API + 부족분 역산 로직

현재 P95 / baseline 비율 → 0~100% 사용률 반환, gap_ratio로 GPU N장 부족 산출

3

대시보드 시각화 + 이메일 경보

게이지 차트 + 시계열 그래프, 50% 초과 시 자동 이메일 알림

산출 공식

사용률 // baseline = RPS 하위 5% 시간대 P95
// saturation = timeout 직전 P95

rate = (current_p95 - baseline)
     ÷ (saturation - baseline)

GPU 사용률(%) = rate × 100
부족분 // 레이턴시 비율로 GPU 부족분 역산

gap_ratio = 1 - baseline ÷ current_p95

부족 GPU = 보유 GPU × gap_ratio

// 예: GPU 8장, P95가 baseline의 1.6배
// gap = 1 - 1/1.6 = 0.375
// 8 × 0.375 = 3장 부족
미래 예측 // RPS 성장 추세로 미래 사용률 예측

future_RPS = current × (미래 사용자 ÷ 현재)

→ 해당 RPS에서의 P95 예측
→ 사용률 & 부족 GPU 산출

PHASE 2 — 향후 고도화

안 2. 대기행렬 실측 기반 정밀 산정

3 / 4

현재 한계와 향후 전환

⚠ 현재: 산술 추정의 한계
문제: vLLM 내부 메트릭에 접근할 수 없는 현재 환경에서는
  • 동시 요청 수(L) = 로그에서 λ × W로 추정
  • 순수 GPU 처리 시간(W0) = 한산 시간대 latency로 추정
  • 대기 시간(Wqueue) = W - W0로 추정

→ 모든 값이 추정치의 추정치이므로, 이론적 정확도가 무색해집니다.
→ 같은 추정 데이터라면, 안 1(레이턴시 비율)이 더 직관적이고 결과도 유사합니다.
✓ 향후: vLLM /metrics 개방 시
vLLM /metrics 엔드포인트가 개방되면 실측 데이터를 직접 읽을 수 있습니다:

  • vllm:num_requests_running → 실제 동시 처리 중 요청 수
  • vllm:num_requests_waiting → 실제 큐 대기 인원 수
  • vllm:gpu_cache_usage_perc → KV cache 사용률

→ 추정 없이 실측 기반으로 "GPU 정확히 N대 부족"을 산출할 수 있습니다.
→ 이 시점에서 안 2가 안 1보다 압도적으로 정확해집니다.
🏦
은행 창구 비유: 은행에 창구가 5개인데 대기줄이 길어져 평균 대기시간이 5분이라면, "창구 2개를 더 열면 대기시간이 1분으로 줄어든다"고 수학적으로 정확히 계산할 수 있습니다. vLLM /metrics가 열리면 GPU도 동일한 계산이 가능합니다.

향후 적용할 공식

Little's Law // vLLM /metrics 실측 시

L = vllm:num_requests_running
Q = vllm:num_requests_waiting

사용률 = L ÷ GPU_capacity
부족 GPU = Q ÷ (L ÷ GPU수)

// 추정 아닌 실측 — 정확도 최상
Erlang-C 미래 예측 // M/G/c 큐: c개 창구, ρ = 이용률

future_λ = current × (미래사용자 ÷ 현재)
W(λ) = EC(c, ρ) ÷ (cμ - λ) + 1/μ

// c를 늘려가며 SLO 달성하는 최소 c 탐색
// → "GPU 정확히 N대 추가하면 목표 응답시간 달성"
