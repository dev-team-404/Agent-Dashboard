기술 상세

사용률 공식 // baseline = 새벽 02~06시 P95 레이턴시
// saturation = timeout 직전 P95

rate = (current_p95 - baseline)
     ÷ (saturation - baseline)

사용률(%) = rate × 100
부족분 공식 // 레이턴시 비율로 GPU 부족분 역산

gap_ratio = 1 - baseline ÷ current_p95

부족 GPU = 보유 GPU × gap_ratio

// 예: GPU 8장, P95가 baseline의 1.6배
// gap = 1 - 1/1.6 = 0.375 → 8 × 0.375
= 3장 부족
미래 예측 // RPS 성장 추세 → 곡선 상 미래 위치

future_RPS = current × (미래 사용자 ÷ 현재)
→ hockey-stick 곡선에서 해당 RPS의 P95 예측

기술 상세

❓
vLLM 메트릭 없이 동시요청·대기비율을 어떻게 아는가?
λ(도착률) = request_logs에서 COUNT(*) / 시간 ← 초당 몇 건 들어오는지
W(체류시간) = request_logs의 latencyMs 평균 ← 한 건이 시스템에 머문 시간
L(동시요청) = λ × W ← GPU 내부를 안 봐도 수학적으로 도출 (Little's Law)

W0(순수 처리시간) = 새벽 한산한 시간의 latencyMs ← 대기줄 = 0이므로 순수 GPU 연산 시간
Wqueue(대기시간) = 현재 W - W0 ← "전체 응답시간에서 순수 처리시간을 빼면 = 줄 선 시간"
Little's Law // 동시 요청 수 = 도착률 × 체류시간
// 둘 다 request_logs에서 바로 산출

λ = COUNT(*) ÷ seconds   ← request_logs.timestamp
W = AVG(latencyMs) ÷ 1000   ← request_logs.latencyMs

L = λ × W = 동시 요청 수
// 예: 초당 5건 × 평균 2초 = 항상 10건이 동시 처리 중
대기 비율 + 부족분 // 새벽(02~06시) latency = 순수 GPU 처리 시간 (대기=0)
W0 = AVG(latencyMs) WHERE hour BETWEEN 2 AND 6

// 현재 latency - 순수 처리시간 = 줄 선 시간
Wqueue = W - W0

// 줄 선 비율이 곧 GPU 부족 비율
부족 GPU = 보유GPU × Wqueue ÷ W0
// 예: 8장, W=3s, W0=2s → 8 × 1/2 = 4장 부족
미래 예측 (Erlang-C) // M/G/c 큐: c개 창구, ρ = 이용률

future_λ = current × (미래사용자 ÷ 현재)
W(λ) = EC(c, ρ) ÷ (cμ - λ) + 1/μ

// c를 늘려가며 SLO 달성하는 최소 c 탐색
장점: 이론적으로 가장 정확 장점: GPU 정수로 산출 단점: KV cache 캘리브레이션 1회 필요

기술 상세

Timeout Rate 산출 // request_logs 에서 직접 집계

timeout_count = COUNT(
  status IN (504, 408, 429)
)
total_count = COUNT(*)

timeout_rate = timeout_count ÷ total_count
부족분 공식 // timeout된 요청 = GPU가 처리 못한 분량

부족 GPU = 보유GPU × t ÷ (1 - t)

// 예: GPU 10장, timeout 10%
= 10 × 0.1 ÷ 0.9 = 1.1장 부족
사용률 (포화 시에만) // 성공 토큰 vs 실패 추정 토큰

success_tok = Σ(성공 요청 토큰)
failed_tok = Σ(실패 요청의 input + 예상 output)

사용률 = success ÷ (success + failed) × 100

⚠ timeout 0% 시 → 사용률 < 100%지만 정확한 값 모름
