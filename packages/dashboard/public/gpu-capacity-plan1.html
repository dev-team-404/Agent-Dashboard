EXECUTIVE SUMMARY

GPU 용량 측정 방안

1 / 7

GPU 내부 메트릭(nvidia-smi) 없이, 기존 요청 로그만으로 GPU 사용률 / 부족분 / 미래 수요를 산출하는 방안입니다.

배경

🎯 문제

현재 GPU 인프라의 실제 사용률을 정량적으로 파악할 수 없어,
"GPU가 부족한가? 얼마나 부족한가? 언제 증설해야 하는가?"에
데이터 기반 답변이 불가능한 상황입니다.

💡 제안

이미 수집 중인 요청 로그(request_logs)의 응답 시간 변화를 분석하여
GPU 사용률을 0~100%로 실시간 측정하고,
포화 전 조기 경보 + 미래 예측까지 제공합니다.

기대 효과

📈 실시간 사용률 대시보드

"현재 GPU 사용률 65%" — 한눈에 파악

⚠️ 조기 경보 시스템

50% 초과 시 자동 알림 — 장애 발생 전 선제 대응

🔮 미래 수요 예측

"사용자 2배 증가 시 GPU 2대 추가 필요" — 예산 수립 근거

💰 추가 비용 없음

기존 로그 데이터만 활용 — 별도 모니터링 도구 불필요

🚗
핵심 아이디어: 새벽에 20분 걸리던 출근길이 40분 걸리면 도로 50% 혼잡.
GPU도 "한산할 때 대비 얼마나 느려졌는가"로 포화도를 측정합니다.

Baseline이란?
사내 서비스 특성상 새벽 트래픽이 없으므로, "최근 30일 중 RPS 하위 5% 시간대의 P95 레이턴시"를 baseline으로 사용합니다. 이는 GPU가 가장 여유로운 상태의 응답 시간으로, 순수 연산 성능을 대표합니다.
대시보드 표시 예시
오전 10시 (업무 시작)
35%
오후 2시 (평균)
65%
오후 4시 (피크)
91%
⚠ 50% 초과 시 "GPU 증설 필요" 자동 알림

산출 공식

사용률 // baseline = RPS 하위 5% 시간대 P95
// saturation = timeout 직전 P95

rate = (current_p95 - baseline)
     ÷ (saturation - baseline)

GPU 사용률(%) = rate × 100
부족분 // 레이턴시 비율로 GPU 부족분 역산

gap_ratio = 1 - baseline ÷ current_p95

부족 GPU = 보유 GPU × gap_ratio

// 예: GPU 8장, P95가 baseline의 1.6배
// gap = 1 - 1/1.6 = 0.375
// 8 × 0.375 = 3장 부족
미래 예측 // RPS 성장 추세로 미래 사용률 예측

future_RPS = current × (미래 사용자 ÷ 현재)

→ 해당 RPS에서의 P95 예측
→ 사용률 & 부족 GPU 산출

GPU 0.3대분 부족. 직원 2배 시 🔴 즉시 증설 단계 진입."

🟠 주의 | 0.3대 부족

기술 상세

Timeout Rate 산출 // request_logs 에서 직접 집계
// ⚠ 현재 timeout은 503으로 기록됨 (선행 수정 후 504)

timeout_count = COUNT(
  status = 504
  // 429(rate limit)는 제외 — 정책 제한이지 GPU 포화 아님
)
total_count = COUNT(*)

timeout_rate = timeout_count ÷ total_count
부족분 공식 // timeout된 요청 = GPU가 처리 못한 분량

부족 GPU = 보유GPU × t ÷ (1 - t)

// 예: GPU 10장, timeout 10%
= 10 × 0.1 ÷ 0.9 = 1.1장 부족
장점: 10분이면 구현 장점: 거짓말 없음 단점: 여유분 측정 불가 단점: 포화 전 조기 경보 없음
🔧 구현 시 선행 수정
1. Timeout status code 분리 (코드 1줄)
  현재: timeout도 statusCode: 503으로 기록 → 진짜 서버 다운과 구분 불가
  수정: timeout 시 statusCode: 504로 변경
2. Rate Limit(429) 제외
  429는 정책적 제한이지 GPU 부족이 아님 → timeout 집계에서 반드시 제외
⚠
안 3의 한계: 🟢 Timeout 0%일 때는 "여유가 얼마나 있는지" 알 수 없습니다. 포화 전의 정확한 % 측정은 안 1이 필요합니다.
